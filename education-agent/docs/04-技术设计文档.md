# 技术设计文档

## 一、技术选型

### 1.1 核心技术栈

| 层次 | 技术 | 版本 | 选型理由 |
|------|------|------|----------|
| 运行时 | Java | 17+ | LTS 版本，生态成熟 |
| 框架 | Spring Boot | 3.5.10 | 主流微服务框架，自动配置 |
| AI 框架 | Spring AI | 2.0 | Spring 官方 AI 集成，统一抽象 |
| AI 框架 | LangChain4J | - | Java LLM 应用框架，链式调用 |
| ORM | MyBatis-Plus | 最新 | 增强 CRUD，分页/逻辑删除/自动填充 |
| 数据库 | PostgreSQL | 15+ | 支持 pgvector 向量扩展 |
| 向量库 | pgvector | - | PG 原生扩展，无需额外组件 |
| 缓存 | Redis | 7+ | 高性能 KV 缓存 |
| 消息队列 | Kafka | 3+ | 高吞吐异步消息 |
| API 文档 | SpringDoc OpenAPI | 最新 | OpenAPI 3.0 自动文档生成 |
| 工具 | Lombok | 最新 | 减少样板代码 |

## 三、核心模块设计

### 3.1 统一响应模型

```java
@Data
public class Result<T> implements Serializable {
    private Integer code;      // 状态码
    private String message;    // 消息
    private T data;            // 数据

    // 静态工厂方法
    static <T> Result<T> ok(T data)
    static <T> Result<T> fail(ErrorCode errorCode)
}
```

### 3.2 全局异常处理

```
异常类型                          HTTP状态码    响应码
────────────────────────────────────────────────────
BusinessException                200          errorCode.code
MethodArgumentNotValidException  400          400
BindException                    400          400
Exception (兜底)                  500          500
```

### 3.4 向量检索设计

**存储方案**: PostgreSQL + pgvector

```
文本 → EmbeddingModel → Vector(1536) → pgvector 列存储
```

**检索方案**:

```
查询文本 → EmbeddingModel → QueryVector(1536)
    │
    ▼
pgvector IVFFlat 索引检索 (cosine 距离)
    │
    ▼
Top-K 相似结果
    │
    ▼
作为 Context 注入 Prompt → LLM 生成回答
```

**索引配置**:

| 参数 | 值 | 说明 |
|------|-----|------|
| 维度 | 1536 | text-embedding-3-small 输出维度 |
| 索引类型 | IVFFlat | 倒排文件索引，适合中等数据量 |
| 距离函数 | cosine | 余弦相似度 |
| lists | 100 | 聚类中心数 |

### 4.4 多模型动态路由

```
用户请求
    │
    ▼
复杂度评估器
    │
    ├──→ 简单问题 (事实查询、简单计算)
    │        └── gpt-4o-mini (低成本、快速)
    │
    ├──→ 中等问题 (知识讲解、解题思路)
    │        └── gpt-4o-mini + RAG 增强
    │
    └──→ 复杂问题 (深度推理、作文批改、多步规划)
             └── gpt-4o (高质量、慢速)
```

**路由策略**:

| 因素 | 权重 | 说明 |
|------|------|------|
| 问题长度 | 30% | 长问题倾向复杂模型 |
| 学段 | 25% | 小学倾向安全模型，大学/成人可用强模型 |
| 知识点深度 | 25% | 涉及多知识点关联用强模型 |
| 用户历史 | 20% | 根据历史交互复杂度调整 |

### 4.5 多模态集成设计

```
多模态输入
    │
    ├──→ 文本输入 → ChatClient → 文本响应
    │
    ├──→ 语音输入 → STT (语音转文本) → ChatClient → TTS (文本转语音)
    │
    ├──→ 图片输入 → 视觉模型 (题目识别/手写识别) → 文本处理 → 响应
    │
    └──→ 混合输入 → 多模态融合处理 → 响应
```

**支持的模态**:

| 输入模态 | 处理方案 | 输出模态 | 适用场景 |
|---------|---------|---------|----------|
| 文本 | 直接处理 | 文本 | 通用对话 |
| 语音 | STT → 文本 | 文本/TTS | 低龄学生、通勤场景 |
| 图片 (印刷体) | OCR → 文本 | 文本 | 拍照识题 |
| 图片 (手写) | 手写识别 → 文本 | 文本 | 作业提交 |
| 混合 | 分别处理 + 融合 | 文本 | 复杂场景 |

## 五、数据库技术设计

### 5.1 数据库选型理由

PostgreSQL + pgvector 相比独立向量数据库的优势：

| 维度 | PostgreSQL + pgvector | 独立向量数据库 |
|------|----------------------|---------------|
| 运维复杂度 | 低，单数据库 | 高，需维护两套系统 |
| 事务一致性 | 强，同一事务 | 弱，需分布式事务 |
| 查询能力 | SQL + 向量混合查询 | 仅向量检索 |
| 适合场景 | 中小规模 (百万级以下) | 大规模 (千万级以上) |

### 5.2 连接池配置

```
HikariCP
├── maximum-pool-size: 20    # 最大连接数
├── minimum-idle: 5          # 最小空闲连接
├── connection-timeout: 30s  # 连接超时
└── idle-timeout: 10min      # 空闲超时
```

### 5.3 分页实现

```java
// MyBatis-Plus 分页插件 (PostgreSQL)
MybatisPlusInterceptor
    └── PaginationInnerInterceptor(DbType.POSTGRE_SQL)
```
